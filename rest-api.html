Topic Overview

REST API & Web Services

Understanding the architectural style of the web: how modern applications communicate using HTTP, resources, and stateless representations.

Web Architecture

1. What is REST?

REST (Representational State Transfer) is not a protocol or a tool, but an architectural style for designing networked applications. It treats server data as Resources that can be created, read, updated, or deleted using standard HTTP methods.

[Image of REST API architecture client server]

Statelessness

The server does not remember the client state between requests. Every request must contain all information (authentication tokens, parameters) needed to process it.

Client-Server

A strict separation of concerns. The frontend (React/Mobile) is decoupled from the backend (Node/Python), allowing them to evolve independently.

2. HTTP Verbs (The Actions)

REST maps standard CRUD operations (Create, Read, Update, Delete) to HTTP methods.

Verb Operation Safety Description
GET Read Safe Retrieve a resource. Should never modify data on the server.
POST Create Unsafe Create a new resource. Not idempotent (sending twice creates two objects).
PUT Update Idempotent Replace an existing resource entirely.
PATCH Update Unsafe Partial update (e.g., only change the email field).
DELETE Delete Idempotent Remove a resource.

3. Status Codes (The Feedback)

The server communicates the result of the request via numerical codes.

2xx Success
200 OK
201 Created
204 No Content
3xx Redirection
301 Moved Perm
304 Not Modified
4xx Client Error
400 Bad Request
401 Unauthorized
404 Not Found
5xx Server Error
500 Internal Error
503 Unavailable
504 Gateway Timeout

4. Anatomy & Best Practices

JSON Response Structure

// GET /users/101
{
  "id": 101,
  "name": "John Doe",
  "role": "admin",
  "links": {
    "self": "/users/101",
    "orders": "/users/101/orders"
  }
}

URI Naming Conventions

Use Nouns (Plural) Correct: /users, /orders
Avoid Verbs Incorrect: /getUsers, /createNewOrder
Use Hierarchy /users/123/orders/456 represents Order #456 belonging to User #123.
